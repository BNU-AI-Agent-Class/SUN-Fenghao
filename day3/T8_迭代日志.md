# T8 · 迭代日志

## 第 1 轮：危机标签正确，但行为仍可能不安全

| 项 | 内容 |
|---|---|
| 发现的问题 | C5“已经准备好且拒绝找人”被标为 `red`，回复却说“不会把你推给现实里的人”；C2、C4 的现实转介也不足 |
| 归到哪维/哪层 | 伦理合规 + 专业有效 / safety_check，一票否决 |
| **诊断根因** | 安全主要依赖模型提示词和模型自报标签；`red` 标签没有绑定到确定性的输出动作，所以出现“判断对、话术错” |
| 改了什么 | 新增独立 `CrisisGuard`：输入信号触发安全模式；模型输出后再按输入、`safety_level` 和 `risk_flags` 复核；自伤/伤人分别使用确定性现实转介 |
| 再测结果 | 23/23 后端测试通过；同一套线上危机测试从存在 3 个关键缺口变为 8/8 PASS |
| 剩余风险 | 词表无法覆盖所有隐喻和方言；应继续扩充对话上下文级判断，并由专业人员复核危机话术 |

## 第 2 轮：工具调用后把整段 JSON 暴露给用户

| 项 | 内容 |
|---|---|
| 发现的问题 | Gemini 有时返回单元素 JSON 数组或残缺 JSON，前端直接显示协议内容，而不是执行工具后给自然回复 |
| 归到哪维/哪层 | 技术稳定性 / C0 协议解析 |
| **诊断根因** | 解析器只覆盖理想 JSON object；解析失败的 fallback 把原始文本当普通 reply，导致内部协议泄漏 |
| 改了什么 | 解析器兼容单元素数组和 code fence；协议形状但不合法时使用安全 fallback，永不把 raw JSON 当 reply；加入回归测试 |
| 再测结果 | `test_singleton_array_c0_executes_tool_instead_of_leaking_json` 与 `test_malformed_protocol_payload_is_never_exposed_as_reply` 均通过 |
| 剩余风险 | 上游模型可能生成新的畸形结构；保留 parse 日志用于继续补样本 |

## 第 3 轮：工具失败后不会像 Agent 一样继续找

| 项 | 内容 |
|---|---|
| 发现的问题 | 7.14 实际聊天中，启明猜错源码/记忆路径后开始解释和推断，没有稳定地“先找目录→换工具→再读取” |
| 归到哪维/哪层 | 技术 + UX / 工具循环 |
| **诊断根因** | 早期实现接近一次规划、一次工具、一次回复；工具错误没有作为 observation 回填给模型继续决策 |
| 改了什么 | 改为最多 3 步 C1 工具循环；每一步把真实结果回填；重复签名、无有效结果或达到上限时强制自然收束；注册 canonical memory/path/search 工具 |
| 再测结果 | 自动测试模拟错误路径，模型第 2 步改用 `memory.read_user_long_term`，第 3 次调用正确回答；没有无限循环 |
| 剩余风险 | 工具描述质量仍影响 Gemini 的选取；服务器低配时多次 LLM 调用会增加延迟 |

> 三轮都遵循“测 → 找根因 → 改 → 用同一输入再测”，没有把“再问一次模型”当成修复。
