# 部署选型决策卡

| 选项 | 优点 | 不足 | 决策 |
|---|---|---|---|
| GitHub Pages + Railway | 和课程样例一致，配置简单 | 多平台、跨域；后台 timer/文件记忆不方便 | 未采用 |
| 单台 Linux + Docker Compose | 环境一致、迁移方便 | 1 GB 内存下同时跑构建/容器更吃资源 | 本地开发可用，线上暂不采用 |
| **Caddy + systemd + 单台 Linux** | 静态前端与 API 同源；内存开销小；timer/文件记忆自然；可完整控制 | 需要自己备份、监控和更新 | **采用** |

## 线上拓扑

- 域名：`dream.qimingbot.com`
- DNS/TLS：Cloudflare
- 回源：Caddy HTTP；服务器 443 保留给已有 Xray 服务
- 前端：`/srv/qiming-agent-core/frontend-dist`
- 后端：systemd `qiming-agent-core.service`，监听 `127.0.0.1:8000`
- 数据：`/srv/qiming-agent-core/data`，部署代码时不覆盖
- 定时任务：heartbeat、tool worker、prompt patch worker 各自使用 systemd timer

## 为什么不用 Docker 作为当前生产入口

项目提供 Docker Compose 作为本地复现方案，但服务器只有 1 GB 内存。生产环境直接用 venv + systemd 更轻，也方便保留用户文件和定时任务。这里不是“不会用 Docker”，而是根据资源和状态持久化方式做的选择。
